## 1 · Orientation — the cockpit (read this screen; drill down only where you don't already own)

**Domain primer** — the acronyms up front, so the rest reads cleanly:
- **DOL** — U.S. Department of Labor.
- **OFLC** — Office of Foreign Labor Certification; the DOL office that publishes the disclosure data.
- **LCA** — *Labor Condition Application*: the filing an employer submits to sponsor an H-1B worker at a stated wage. A **certified** LCA = the employer is cleared and intends to sponsor. **This is the core signal.**
- **PERM** — the *green-card*-stage labor certification; a different, later step. Wrong layer here; the code guards against loading a PERM file by mistake.
- **PW_WAGE_LEVEL** — Prevailing Wage Level (I–IV). **Level I = entry wage**, the new-grad signal the whole thing filters on.
- **SOC code** — Standard Occupational Classification; the federal job-category code. Design codes used: `15-1255` (Web & Digital Interface = UI/UX), `27-1024` (Graphic), `27-1021` (Commercial & Industrial).
- **H-1B** — the temporary skilled-worker visa the LCA underlies (the sponsorship-intent signal).
- **OPT / STEM-OPT** — Optional Practical Training: post-graduation work authorization for international students (STEM extends it). Design is likely *not* STEM-eligible, so a shorter runway.
- **CASE_STATUS = Certified** — the filing DOL approved; the only rows the engine keeps.
- **Layer A** — the coined name for the **OPT-now postings pipeline** (auto-classify which shortlist companies have a live entry-level design req *right now*). **Deliberately cut to v2** (`decision_log` #8); v1 replaces it with a manual "are they hiring now?" eyeball. Named here on first use because the rest of the docs use the shorthand.
- **iGavel** — the FY2025-Q4 top single-quarter design sponsor, hardcoded as the verification "golden anchor" (see D3).

**What it is.** A grounded **visa-sponsorship diagnostic** for **international new-grad designers who need
US work sponsorship** (validated so far on one real designer). It reads mandated U.S. Department of Labor
filings (the **LCA** disclosure data) to find companies that *actually* sponsor designers at the **entry
wage** (`PW_WAGE_LEVEL = I`, under three design SOC codes), ranked by how *repeatedly* they sponsor across

*Three data inputs converge: one automated + deterministic (the DOL grounding), two manual (the applicant's

**Input 1 — DOL LCA data (grounding; automated).**

| Stage | Data + form | Source | → becomes | By |
|---|---|---|---|---|
| Retrieve | one `.xlsx` per fiscal quarter, ~80–144 MB, ~98 cols, named `LCA_Disclosure_Data_FY2025_Q4.xlsx` | downloaded **by hand** from the DOL OFLC performance-data page (dol.gov); free, public, mandated → `data/raw/` (gitignored) | — | you |
| Narrow | the 12 `REQUIRED_COLUMNS` (case status, visa class, SOC code/title, employer, worksite, wage rate/unit, `PW_WAGE_LEVEL`), typed | resolved **by name** off the file's header (column order isn't stable across quarters) | `data/processed/lca_<q>.parquet` (small typed cache) | `xlsx_to_parquet` (`convert_quarters.py`) |

**How are roles found?** There's no job-title matching — **the SOC code *is* the role.** Filtering the
certified filings to the three design SOC codes at wage-level I *is* how sponsor-viable roles are found; the
employer is whoever filed them.

**Input 2 — the applicant's info (manual).** Format = **text + links pasted by hand** into the
`[[PORTFOLIO]]` block of `prompts/gap_read.md` (case studies, roles, tools, years). A **resume** is the
weaker fallback (less signal). The code never parses a portfolio file — it's a prompt input. *(Manual-input
mechanics: the prompt has four bracketed blocks — `[[PORTFOLIO]]`, `[[SHORTLIST]]`, `[[LIVE_POSTINGS]]`,
`[[TARGET_ROLE]]` — that you replace by hand before pasting into your own LLM; nothing auto-fills them.)*

**Input 3 — live postings (manual).** For ~5–8 shortlist companies you paste one real current entry-level
design posting each into `[[LIVE_POSTINGS]]` — what makes the advice named-company-specific rather than
generic. You gather these yourself from careers pages / LinkedIn.

**The merge — what receives all three and evaluates them.** The **portfolio review** (`prompts/gap_read.md`,
run in your own Claude/ChatGPT, human-reviewed) takes the portfolio (Input 2) + shortlist rows pasted from
`sponsors_levelI.csv` (Input 1; prefer repeat-sponsors + role-matched) + the live postings (Input 3) + the
target role, and **relates the portfolio to what the named companies' real postings ask for** → 3–5 gaps
(each tied to a named company) → exactly 3 portfolio projects (each targeting 1–2 named companies, closing a
named gap, 2–4 weeks, a specific artifact). Reviewed output → `output/private/gap_read_filled.md`.

**Final output.** `build_report.py` reads `sponsors_levelI.csv` + `gap_read_filled.md` (if present) → one

**Two boundaries.** *Provenance:* every shortlist cell traces to a source file + quarter (the
`build_shortlist` provenance block prints it). *Public/private:* the DOL data + method + employer shortlist
are public record; the applicant's portfolio and the final report are private (`output/private/`,
gitignored).

<details>
<summary><b>A · The engine</b> — <code>engine/sponsors.py</code> · the deterministic Layer-1 core</summary>

**Role.** The whole "what the product *is*" on the data side: turn raw DOL filings into the grounded sponsor
shortlist. It's the reusable, role-parameterized function a future self-serve UI would call — it returns
DataFrames and imports nothing about HTML/PDF (that separability is half of D4).

**How it works (first-principles, three stages).**
1. **Ingest** — `xlsx_to_parquet` streams a 100+ MB raw xlsx and writes a narrow 12-column parquet cache
   (D1). Runs once per quarter.
2. **Filter** — `load_certified_rows` reads the parquet(s) and keeps rows that are
   `CASE_STATUS == CERTIFIED` **∩** `SOC7 ∈ role's SOC set` **∩** `PW_WAGE_LEVEL ∈ Level-I spellings`, drops
   blank-employer rows, and derives `EMP_NORM` / `SOC7` / `WAGE_ANNUAL`. Output = one row per certified
   filing (this is the grounding the portfolio review consumes).
3. **Aggregate** — `build_sponsor_table` groups those rows by `EMP_NORM`, counts filings, counts distinct
   quarters, rolls up SOC titles / states / cities / wage min-median-max, and **sorts by `quarters_present`
   then `filing_count`** (D5). Output = one row per employer = the shortlist. Role generality lives in
   `ROLE_SOC` (`:32`) — adding "consultant" later is one dict entry (D4/#7).

**Key decisions inside A.**

**▸ D1 — xlsx→parquet: streaming openpyxl, by-name 12-col select · Open · full · maint·cost · `[code]`**
- **The call.** `engine/sponsors.py:134` `xlsx_to_parquet`: open the raw xlsx with
  `openpyxl(read_only=True)`, read the header, **resolve the 12 `REQUIRED_COLUMNS` by name** (not index),
  pull only those, coerce wages numeric / rest to string, write a narrow parquet; downstream reads parquet
  only.
- **Why (tied to role + lens).** `read_only=True` streams rows lazily (the workbook XML is never built into
  objects). **By-name** resolution survives that DOL's *column order is not stable across quarters* (`[doc]`
  B4 records a real index bug this caught: FY2025 Q1 has `WORKSITE_COUNTY` where Q4 has `WORKSITE_CITY`) —
  that's the **maintainability** win. Parquet then gives typed, fast repeat reads so the slow parse happens
  once — the **cost** win.
- **Option landscape (on the axes).**

  | option | peak memory | speed | deps | schema-drift | verdict |
  |--------|-------------|-------|------|--------------|---------|
  | `pandas.read_excel` | ✗ high (loads all) | slow | pandas | by-name ok | OOM risk on 144 MB |
  | **stream openpyxl (chosen)** | ~ok | slow-ish | boring/already-implied | by-name ✓ | rows still materialized (see below) |
  | `python-calamine` | low | **fast** (Rust) | newer dep | by-name ✓ | big speedup, v2 candidate |
  | `polars.read_excel` | low | fast | new dep | ✓ | also calamine-backed |
  | `duckdb` | low | fast | new dep | query | better for the v2 query path |
- **Why it wins here.** A handful of one-off quarterly parses → "low memory + boring already-needed deps +
  robust to DOL column churn" beats raw speed. `[likely-why]`
- **Would you make this call again?** Yes for v0. `[reconsider]` Two catches: (a) `:159`
  `rows = [tuple(r[i] for i in idx) for r in ws.iter_rows(...)]` materializes **every** row into a Python
  list before the DataFrame — it streams the parse but **not memory**; profile it on the 144 MB quarter. (b)
  For the v2 automated refresh, swap to calamine/polars + duckdb.
- **Rebuild path.** Read one 100 MB quarter three ways (`read_excel`, stream-openpyxl, calamine), record
  peak RSS + wall time; then re-read only 12 cols via calamine and feel the speedup. You'll own *why*
  openpyxl is the safe-not-fast pick and exactly what you'd trade to switch.
- **Firsthand docs.** openpyxl read-only mode (`openpyxl.readthedocs.io/en/stable/optimized.html`) ·
  `python-calamine` (github.com/dimastbk/python-calamine) · Polars `read_excel`
  (docs.pola.rs) · Arrow/Parquet columnar format (parquet.apache.org/docs).

**▸ D2 — Employer normalization: rule-based, conservative under-merge · Open · full · UX·maint · `[code]`**
- **The call.** `engine/sponsors.py:94` `normalize_employer`: uppercase → strip punctuation → drop trailing
  corporate suffixes (`_CORP_SUFFIXES`) → drop a leading `THE`; `None` for blank names (so they group and
  get excluded). `build_sponsor_table` groups on this `EMP_NORM`.
- **Why (tied to role + lens).** Collapses spelling/entity variants ("The Deloitte Consulting, LLP." +
  "Deloitte Consulting LLP" → `DELOITTE CONSULTING`) so one employer = one row and the filing count is real.
  Deliberately **conservative**: it won't merge genuinely different legal spellings — because a **false merge
  misleads the user's career bet (UX/correctness)**, which is worse than splitting a count slightly.
  Over-merge is the expensive error, so it under-merges on purpose.
- **Option landscape.** exact match (splinters variants) · **rule-based suffix strip (chosen)** (precise,
  explainable; misses `IBM`↔`International Business Machines`) · fuzzy/`rapidfuzz` (more recall, risks false
  merges) · record-linkage/`dedupe` (overkill at this scale) · canonical key (EIN/DUNS) · small curated
  alias map (cheap recall boost).
- **Why it wins here.** The shortlist is tens–low-hundreds of employers you can eyeball, and it underwrites a
  real career bet → **precision + explainability > recall**. `[doc]`
- **Would you make this call again?** Yes for v0 at this scale. `[reconsider]` Add a small curated alias map
  before reaching for fuzzy; revisit only if the list outgrows eyeball size or feeds an automated product.
  Also watch the `CO`/`COMPANY` suffix strip — it will quietly mangle a firm literally named "…Co".
- **Rebuild path.** Normalize 30 real `EMPLOYER_NAME` values by rule; run `rapidfuzz` at ratio 90 then 80
  and watch a false merge appear; add a 5-entry alias map and see recall rise without the false-merge risk.
  You'll own the precision/recall dial.
- **Firsthand docs.** `rapidfuzz` (maxbachmann.github.io/RapidFuzz) · Python `dedupe`
  (docs.dedupe.io) · pandas `groupby` (pandas.pydata.org/docs/reference/groupby.html).

**▸ D5 — Full-year roll-up + repeat-sponsor ranking · OWNED · confirm · UX · `[doc]`**
`build_sponsor_table` sorts by `quarters_present` first (`:277`), then `filing_count`. The claim "*actually*
sponsors entry-wage designers" is far stronger for a company seen across multiple quarters than a single
spike — that's the **UX/quality** of the shortlist the user reads. Because the engine is quarter-parameterized,
full-year was a one-call extension, not new scope (single quarter stays the documented kill-test).
`decision_log` B2. Owned.

**▸ D6 — Level-I spelling tolerance + wage annualization · OWNED · confirm · maint · `[code]`**
`LEVEL_I_VALUES = {"I","LEVEL I","1"}` (`:66`) tolerates how `PW_WAGE_LEVEL` is spelled across DOL vintages
(the comment says `value_counts()` before trusting a new vintage; FY2025 uses bare `"I"`). `_annualized_wage`
multiplies `WAGE_RATE_OF_PAY_FROM` by a unit factor (`_ANNUALIZE`, `:77` — HOUR=2080, YEAR=1, MONTH=12, …) so
wages compare across HOUR/YEAR/etc. — explicitly a *secondary* display signal, not a filter. Small
data-vintage robustness calls; owned. **Watch:** an unknown wage unit silently yields `None` (acceptable,
since wage is secondary).

**Growth (what A's shape enables/constrains).** Role-parameterization + report-agnostic return make new roles
and a future query UI cheap (load-bearing, and it's the right load-bearing). The constraint that will bite
first is D1's in-memory materialization if this ever runs at automated full-corpus scale.

<details>
<summary><b>B · The verification gate</b> — <code>engine/verify.py</code> · the raise-and-stop "cell"</summary>

**Role.** Make it impossible to *silently* ship a bogus shortlist someone bets a career on. A failed check
raises `VerificationError` and stops the run before any CSV is written.

**How it works.** `run_all` runs six checks in order: `verify_normalization_unit` (data-independent, 4 fixed
cases at `:32-37`), `verify_columns`, `verify_nonempty`, `verify_count_consistency` (Σ filing_count ==
selected rows), `verify_collapse` (a multi-spelling employer collapses to one row, else "grouping reduced
rows"), `verify_golden_killtest` (top kill-test sponsor present). Five are **quarter-independent invariants**
(they hold for any quarter); one is a **golden anchor**.

**Key decision inside B.**

**▸ D3 — Verification = invariants + one golden anchor · REVISIT · full · maint · `[code]`**
- **Why (lens).** Invariants (units, schema, count-consistency, collapse) travel across quarters →
  **maintainable** across DOL's churn; the golden anchor catches "the parse silently drifted" that pure
  schema checks miss. `[doc]` (B4: an assert caught the real index bug mid-build.)
- **Option landscape.** no checks · ad-hoc prints · **invariant asserts + one golden anchor (chosen)** ·
  `pytest` suite · schema validation (`pandera`) · property-based (`hypothesis`). The chosen mix is
  invariant-heavy (robust) with a single sharp drift detector.
- **The residual (why REVISIT).** `verify_golden_killtest` defaults to `IGAVEL` (`engine/verify.py:86`), an
  FY2025-**Q4** employer, and `build_shortlist` calls `run_all` against *whatever* `--quarters` you pass,
  with **no override** (`build_shortlist.py:65`). Run a quarters-subset without iGavel and it raises on
  **correct** data — the prior build's exact failure mode, now shrunk to 1 of 6 checks. **Fix (shape):** add
  a `killtest_top`/fixture-quarter parameter through `run_all` → `verify_golden_killtest`, and in
  `build_shortlist` assert the golden anchor **only when its fixture quarter (FY2025Q4) is in `quarters`**;
  otherwise pass on invariants + schema only. Two call sites (`verify.py:92`, `build_shortlist.py:65`) — you
  open both files to write it. (First Change #1 below.)
- **Rebuild path.** Add a `pandera` parquet schema + a `hypothesis` idempotence test for `normalize_employer`;
  then make the golden check take a fixture quarter and feel the difference between "input changed" and
  "parse drifted."
- **Firsthand docs.** `pandera` (pandera.readthedocs.io) · `hypothesis` (hypothesis.readthedocs.io) ·
  `pytest` (docs.pytest.org).

**Growth.** Invariant-first design is CI-ready as-is; the golden anchor is the one piece not yet safe to run
on arbitrary input. Fixing D3 unlocks "run any quarter/subset in CI."
